Beatly Documentation

Deploy-ready overview of the current product

Beatly combines a patient-facing React application, a FastAPI cardiovascular scoring engine, and a lightweight Node orchestration layer for grounded intake phrasing, rich reports, and assistant responses.

The key design rule remains: diagnosis and scoring stay in the ML/rule path. LLMs are used only for explanation, question phrasing, and narrative presentation after the clinical engine has already produced findings.

Diagnosis path ML + rules, not LLM
Interface languages English, Hindi, Hinglish
Generation layer Intake wording, reports, assistant chat
Frontend status Adaptive intake, loading shells, section audio

1. System Layout

Three-tier product architecture

Client

React + Vite application responsible for intake, clinical input, result tabs, audio playback, and chat UI.

  • `client/src/App.jsx` controls the main state machine.
  • `client/src/questionEngine.js` decides what to ask next.
  • `client/src/ReportNarratives.jsx` renders doctor-style report sections.
  • `client/src/i18n.js` handles English, Hindi, Hinglish, and mojibake repair.

Node server

Acts as the public API layer, forwards scoring calls to FastAPI, and handles grounded narrative generation.

  • `server/index.js` serves `/api/*` routes.
  • Uses Groq for intake-question, report, assistant, and insight generation.
  • Prompts explicitly forbid diagnosis invention.

ML engine

FastAPI service that owns medical scoring, thresholds, refinement, and history persistence.

  • `ml_engine/app/main.py` exposes predict, refine, simulate, history, and health routes.
  • `ml_engine/app/models.py` contains the risk engine.
  • `ml_engine/app/clinical_interpreter.py` and `config.py` contain rule logic.

Fallback behavior

When the live path fails, the client can still produce a usable assessment and narrative shell.

  • `client/src/fallbackEngine.js` mirrors key scoring and result behavior.
  • Fallback is for resilience, not for claiming the same confidence as a validated clinical backend.
2. Request Flow

How the product moves from intake to explanation

Patient opens app
  -> chooses English / Hindi / Hinglish
  -> answers adaptive intake questions
  -> optional lab values added
  -> client sends payload to /api/predict
  -> Node server forwards to FastAPI
  -> ML + rules return scored cardiovascular result
  -> client requests /api/report for grounded narrative sections
  -> client renders confidence-rated tabs
  -> optional /api/assistant answers explain the existing findings
  -> optional doctor deep dive uses post-result follow-up refinement
Core scoring and clinical interpretation happen before the LLM layer is asked to write anything.
3. Language Behavior

English, Hindi, and Hinglish modes

English

Default interface, narration, and assistant mode.

Hindi

Expected to render in Devanagari. Prompts explicitly instruct the server to avoid Roman Hindi.

Hinglish

Roman-script hybrid mode for users who want conversational Hindi without Devanagari.

Legacy text repair

The localization layer repairs legacy UTF-8 / Latin-1 mojibake so older broken strings do not leak into the interface.

4. Reporting Model

How the result tabs are generated

After the clinical engine returns risk data, the app requests a structured JSON report from the Node layer. The prompt tells the model to write only grounded, medically useful sections and omit weak filler.

Layer Role
FastAPI result Risk score, confidence, patterns, interpretations, care flags, food guidance, follow-up anchors
Node prompt Transforms those findings into polished tab sections with `confidence_score` and `confidence_note`
Client rendering Shows only materially useful sections and gives each one its own listen button
The report layer should explain the output, not replace the diagnostic engine.
5. API Surface

Routes in the current stack

GET  /api/health
GET  /api/history
POST /api/predict
POST /api/refine
POST /api/simulate
POST /api/insights
POST /api/intake-question
POST /api/report
POST /api/assistant

`predict`, `refine`, and `simulate` are clinical/scoring paths. `intake-question`, `report`, `assistant`, and `insights` are grounded language-generation paths.

6. Deployment

Recommended production setup

Frontend

npm install
npm --workspace client run build

Serve `client/dist` from your web tier.

Node server

npm --workspace server run start

Set `PORT`, `ML_ENGINE_URL`, `GROQ_API_KEY`, and optionally `GROQ_MODEL`.

FastAPI service

python -m pip install -r ml_engine/requirements.txt
python -m uvicorn ml_engine.app.main:app --host 0.0.0.0 --port 8000

Deploy this behind an internal URL and point the Node layer to it.

Routing

Serve the frontend on the main domain and reverse proxy `/api/*` to the Node server.

7. Operational Notes

Safety, audio, and release guidance

  • Keep provider keys on the server only. If a key was previously exposed in the frontend, rotate it.
  • Speech quality depends on browser and OS voices. Hindi playback is best when a real `hi-IN` voice is installed.
  • Result sections are confidence-rated, but this still does not make the product a clinically validated medical device.
  • Before public release, the team should still run a full build, environment verification, and endpoint smoke test.
